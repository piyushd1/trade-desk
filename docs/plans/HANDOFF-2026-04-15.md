## Known risks to watch for

### 1. Redis connection error at backend startup (HIGH)

`docker-compose.prod.yml` does NOT include a Redis service (deferred to Phase 1.5 per plan). If the backend code hard-imports Redis at module load time or eagerly connects at startup, the container will crash-loop with something like `redis.exceptions.ConnectionError: Error connecting to localhost:6379`.

**Fix if it happens**: add a tiny Redis service to `docker-compose.prod.yml`. Add this service block:

```yaml
  redis:
    image: redis:7-alpine
    container_name: trade-desk-redis
    restart: unless-stopped
    platform: linux/arm64
    mem_limit: 100m
    mem_reservation: 50m
    command: redis-server --save "" --appendonly no
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 30s
      timeout: 5s
      retries: 3
```

Then update the `backend` service environment to override `REDIS_URL` to `redis://redis:6379/0` (using the compose network service name, not `localhost`). Add to the backend's `environment:` block:
```yaml
      - REDIS_URL=redis://redis:6379/0
      - CELERY_BROKER_URL=redis://redis:6379/1
      - CELERY_RESULT_BACKEND=redis://redis:6379/2
```

And add to the `backend` service:
```yaml
    depends_on:
      redis:
        condition: service_healthy
      # keep any existing depends_on entries
```

Commit + push + `git pull` on Pi + `docker compose up -d --build backend redis`.

### 2. Tailscale serve path routing (MEDIUM, unverified)

The `trade-desk-tailscale-serve.service.template` uses:
```
ExecStart=/usr/bin/tailscale serve --bg --https=443 --set-path=/api http://localhost:8000
ExecStart=/usr/bin/tailscale serve --bg --https=443 http://localhost:3001
```

Per Tailscale docs, mount points "behave similarly to Go ServeMux", which implies the full request path is preserved when forwarded. So `/api/v1/foo` on the tailnet should reach the backend as `/api/v1/foo` (matching the FastAPI `api_router` prefix).

**If the backend receives `/v1/foo` instead of `/api/v1/foo`**, `tailscale serve` IS stripping the `/api` prefix before forwarding. Fix: change the target URL to include the prefix:
```
ExecStart=/usr/bin/tailscale serve --bg --https=443 --set-path=/api http://localhost:8000/api
```
Then `sudo systemctl restart trade-desk-tailscale-serve.service`.

### 3. Permission denied on /app/data or /app/logs volumes (LOW)

Named volumes can have root ownership when first created. The Dockerfile chowns `/app` to `tradedesk:tradedesk` at build time, but that's inside the image — when Docker creates a volume and mounts it over `/app/data`, the mount point may revert to root-owned.

**Fix if it happens**: `docker compose exec -u root backend chown -R tradedesk:tradedesk /app/data /app/logs`
