When actually running Stage 3 end-to-end on the Pi, a set of bugs surfaced that weren't visible in Mac-side dry runs. All fixed and committed:

- **`6e2a533` — fix(deploy): Stage 3 Pi runtime fixes**
  - `backend/app/database.py`: branch connection-pool config on the actual `DATABASE_URL` dialect instead of `APP_ENV`.   - `deployment/scripts/setup-pi.sh`: detect `SUDO_USER` so `sudo bash setup-pi.sh` resolves to the real invoking user instead of root. Uses `getent` for group + home lookup.
  - `deployment/systemd/trade-desk-tailscale-serve.service.template`: the backend target URL must be `http://localhost:8000/api`, not `http://localhost:8000`. `tailscale serve --set-path=/api` DOES strip the mount prefix before forwarding — we verified with a `/api/v1/auth/login` probe that returns 405 (route exists, method disallowed) vs 404 (route not found).

- **`d58cdf1` — fix(zerodha): complete OAuth flow end-to-end**

## Architectural decisions locked in (see plan file for rationale)

- **Tailscale serve** for TLS + path routing (NOT Caddy or nginx). `/api` → backend:8000, `/` → frontend:3001.
- **SQLite** via `sqlite+aiosqlite:////app/data/tradedesk.db` in named Docker volume `tradedesk-data`. DATABASE_URL is overridden in `docker-compose.prod.yml` environment block; the value in `.env` doesn't apply inside Docker.
- **SQLite WAL mode** applied on every connection via event listener in `backend/app/database.py`.
- **`uvicorn --workers 1`** pinned in `Dockerfile.backend` CMD because the Stage 4 APScheduler must be a single in-process instance.
- **No Redis/Celery** in Phase 1 (deferred to 1.5 if APScheduler is insufficient).
- **3 systemd units on the Pi**: `trade-desk-migrate.service` (oneshot), `trade-desk.service` (main), `trade-desk-tailscale-serve.service` (oneshot).
- **Alembic migrations** run via the dedicated oneshot service, NOT auto-on-container-start.

## Known limitations & gotchas (verified during deploy)

All three "risks" from the original version of this doc turned out to be either non-issues or already fixed. Keeping this section as institutional memory.

### 1. Redis connection at startup — NOT a real issue

Original concern was that the backend might hard-import Redis and crash. In practice, backend startup on the Pi (with `APP_ENV=production`, no Redis service in compose) runs cleanly to `Application startup complete` — see `/backend/app/main.py` `lifespan()`. Redis and Celery are mentioned in `.env` defaults but nothing at module load eagerly connects. If Phase 1.5 ever adds a real Redis dependency, add a service to `docker-compose.prod.yml`; for now, no action.

### 2. Tailscale serve `/api` prefix stripping — FIXED

`tailscale serve --set-path=/api http://localhost:8000` **does** strip the `/api` prefix before forwarding. Verified empirically: a request to `https://.../api/v1/auth/login` reached the backend as `/v1/auth/login` and 404'd. Fix (now in the committed template): target URL must be `http://localhost:8000/api` so the prefix is re-added. Confirmed working via a 405 response on `/api/v1/auth/login` (route exists, GET not allowed, POST-only).

### 3. Named volume permissions — FIXED permanently

`tradedesk-data` first-mount landed as root-owned, blocking the backend container from opening SQLite. Runtime fix on the Pi: `docker compose run -u 0 ... chown -R tradedesk:tradedesk /app/data /app/logs`. Permanent fix in `Dockerfile.backend` (Phase A2 commit): `RUN mkdir -p logs data && chown -R tradedesk:tradedesk /app`. Fresh volumes on any future machine will inherit correct ownership.

### 4. Zerodha daily re-authentication — Kite retail-app limitation, not a bug

Zerodha access tokens expire at **6:00 AM IST daily**, hard cutoff. Retail Kite Connect apps do not receive `refresh_token`s (only premium/partner apps do). The `token_refresh_service` at `backend/app/services/token_refresh_service.py:185-216` filters for `refresh_token_encrypted.isnot(None)` and silently finds nothing to refresh — effectively dead code for this deployment. The "Refresh Token" button in Settings calls the same path and also fails silently.

**Practical impact**: user must click "Connect to Zerodha" once per morning after 6 AM IST. OAuth round-trip is ~10 seconds. This is correct behavior per the Kite Connect v3 contract, not a TradeDesk bug. Phase F (frontend rebuild, deferred) will surface a proactive expiry banner; Phase B10 gates the dead refresh-service loop.

### 5. Docker `mem_limit` discarded on Pi OS bookworm — cosmetic

`docker compose run` prints:
```
Your kernel does not support memory limit capabilities or the cgroup is not mounted. Limitation discarded.
Your kernel does not support memory soft limit capabilities or the cgroup is not mounted. Limitation discarded.
```
This is a cgroup v1/v2 transition artifact on Pi OS bookworm and is **cosmetic**. The `mem_limit` / `mem_reservation` values in `docker-compose.prod.yml` are advisory in this kernel and get silently dropped. Containers still run within host memory constraints. Not a blocker; worth knowing so you don't chase it.

### 6. Admin user must be seeded manually

`backend/scripts/create_default_user.py` is NOT invoked by Alembic migrations or `lifespan()`. After a fresh DB (fresh `tradedesk-data` volume), run:

```bash
docker compose -f ~/trade-desk/deployment/docker/docker-compose.prod.yml exec -T backend python scripts/create_default_user.py
```

It reads `ADMIN_USERNAME` / `ADMIN_EMAIL` / `ADMIN_PASSWORD` from `backend/.env` (already loaded as env vars in the container) and creates a single admin user with bcrypt-hashed password. Log in with `ADMIN_USERNAME` + the plaintext `ADMIN_PASSWORD` at `https://tradedesk.tail1ad4b2.ts.net/`.
